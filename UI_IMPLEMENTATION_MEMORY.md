# 山海灵感便利店 UI 实现记忆

> 用途：记录已经出现过的问题、确认过的设计原则和每次 UI 修改后的验收要求。后续开发前先阅读，避免重复踩坑。
>
> 最后更新：2026-06-29

## 1. 视觉基准

- 唯一视觉基准是用户提供的 UI 图：`ChatGPT Image 2026年6月25日 09_40_56.png`。
- 产品定位是移动端“未来感便利店”，不是通用商城模板。
- 核心视觉关键词：电光蓝、白色内容面、紧凑卡片、细线图标、圆角手机界面、山海神兽 IP。
- 页面优先按 `390 × 844` 移动视口设计，同时保证桌面端以手机外壳居中展示。
- 桌面展示背景可以使用深蓝氛围，但不能影响手机内部的白色页面层级。

## 2. 已出现的问题

### 2.1 中文编码损坏

**现象：** 导航、商品名、按钮和页面标题出现乱码，例如 `灞辨捣`、`璐墿`。

**原因：** UTF-8 中文被按错误编码读取或写回。

**约束：**

- 所有源码和 Markdown 文件必须使用 UTF-8。
- 修改后用下面的命令检查常见乱码字符：

  ```powershell
  rg -n "灞|鐏|璐|鈥|馃|楼|脳" src -g "*.ts" -g "*.tsx"
  ```

- 不要复制终端中已经乱码的文本回源码；应从 UI 稿或正确语义重新录入。

### 2.2 图标体系不统一

**现象：** 页面混用了 emoji、文字字符、自绘 SVG 和不同风格的图标，与 UI 稿差异明显。

**约束：**

- 功能图标统一使用 `lucide-react`。
- 常规图标采用 `1.5–1.8` 的 `strokeWidth`，选中态可提高到 `2.4–2.5`。
- 禁止用 emoji 充当分类、导航或操作图标。
- 禁止用 `+`、`✓`、`♡` 等字符替代已有的 Lucide 图标；应使用 `Plus`、`Check`、`Heart` 等组件。
- 底部导航固定使用：首页 `House`、分类 `LayoutGrid`、发现 `CircleCheckBig`、购物车 `ShoppingCart`、我的 `UserRound`。

### 2.3 视觉风格偏离 UI 稿

**现象：** 旧版使用蓝紫渐变、过度发光、过大的圆角和宽松留白，更像概念页面而不是参考稿中的便利店 App。

**约束：**

- 主色统一为 `#075cff`，深色辅助色为 `#003ec8`。
- 浅色页面背景为 `#f5f7fb`，卡片使用纯白。
- 卡片常用圆角为 `12–16px`；Banner 可使用 `18–22px`；不要所有元素都使用超大圆角。
- 阴影要轻：用于区分白卡片与浅灰背景，不制造悬浮玻璃效果。
- 商品价格默认使用深色或主蓝色，不沿用旧版大面积朱红色。
- 页面信息密度应接近 UI 稿：标题约 `15–16px`，商品名约 `11–12px`，辅助文字约 `8–10px`。
- 登录页白色主按钮必须使用独立 `.login-primary` 样式，并显式设置蓝色文字；全局 `a { color: inherit }` 会覆盖普通文字颜色工具类。
- 登录页神兽模型使用外层容器承担位移动画，图片自身只处理亮度呼吸，避免动画 `transform` 覆盖水平居中变换。

### 2.4 商品素材与 UI 稿不完全一致

**现象：** 当前本地素材与 UI 稿中的包装图并非逐像素相同，部分商品共用了图片。

**约束：**

- 优先复用 `src/assets/` 中现有的山海主题商品图，不用网络占位图。
- 商品卡片使用一致的 `object-cover` 和固定尺寸，避免原图比例导致布局跳动。
- 如果用户提供新的单品切图，应替换素材而不是通过 CSS 勉强模拟。
- 未获得透明底素材时，不承诺与 UI 稿中的商品抠图效果完全一致。

### 2.5 固定底栏遮挡内容

**风险：** 底部导航、购物车结算栏或商品详情操作栏为绝对定位，可能遮挡滚动内容。

**约束：**

- 显示底部导航的页面必须保留至少 `82px` 的底部内容空间。
- 结算栏和商品详情操作栏应与普通底部导航互斥。
- 在 `390 × 844` 和窄屏手机视口下检查最后一张卡片能否完整滚动出来。

### 2.6 返回按钮路由错误

**风险：** 所有返回按钮固定跳转首页，会破坏“商品详情 → 分类 → 返回”的浏览路径。

**约束：**

- 通用 `TopBar` 返回操作使用浏览器历史 `history.back()`。
- 只有业务明确要求返回固定页面时，才使用固定 `Link`。

### 2.7 构建环境差异

**现象：** 直接执行 `pnpm run build` 可能触发依赖目录重建，并因无 TTY 或依赖构建脚本审批而中断。

**当前可用验证命令：**

```powershell
& 'C:\Users\19617\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\bin\node.exe' node_modules\vite\bin\vite.js build

& 'C:\Users\19617\.cache\codex-runtimes\codex-primary-runtime\dependencies\node\bin\node.exe' node_modules\typescript\bin\tsc --noEmit
```

- 不要为了绕过问题随意删除 `node_modules` 或锁文件。
- 项目同时存在 `pnpm-lock.yaml` 和 `bun.lock`，更新依赖前先确认团队实际使用的包管理器。

### 2.8 uni-app H5+ 平台能力调用

**原则：** 项目的目标运行容器是 uni-app H5+。安全区、缓存和原生权限等平台能力必须通过 `uni` / H5+ API 实现，不能直接使用浏览器 API 代替。

**约束：**

- 全局平台工具统一放在 `src/utils/`，业务页面不直接散落原生平台判断。
- 权限统一从 `src/utils/permission.ts` 聚合调用；Android、iOS 与 PC 浏览器实现分别放在 `android.ts`、`ios.ts`、`pc.ts`。
- PC 浏览器权限通过 Permissions API 查询；没有对应浏览器能力的 App 接口必须在 `pc.ts` 提供同签名空实现，不能在业务页面散落平台分支。例如 PC 端“清除缓存”为空操作。
- 刘海屏安全区优先通过 `uni.getWindowInfo()` 获取，旧运行时回退到 `uni.getSystemInfoSync()`；读取 `safeAreaInsets` / `safeArea` 并同时处理顶部与底部安全距离。
- 安全区高度必须使用运行时返回的真实值，禁止设置固定最小高度；PC 端安全区为 `0` 时不得出现全局顶部留白。
- “清除缓存”使用 `uni.getStorageInfo` 和 `uni.clearStorage`，禁止改用 `window.caches`、`localStorage.clear()` 或 `sessionStorage.clear()`。
- SSR 或普通浏览器预览中必须先判断运行时 API 是否存在，不能因顶层访问 `uni` / `plus` 导致页面崩溃。
- “我的”不展示“发票管理”；“设置”不展示“设备管理”。

### 2.9 H5+ 静态包复制后空白

**现象：** `dist/client/` 中存在 `index.html`，但复制到 H5+ 项目或子目录运行后页面空白。

**原因：** TanStack Start SPA shell 中的入口脚本可能生成为 `/./assets/*` 绝对路径，复制后会从错误的容器根目录加载；Browser History 在无服务端回写的本地容器中也无法稳定处理页面路径。

**解决：**

- 客户端统一使用 Hash History，页面路径采用 `#/home` 等形式。
- `npm run build` 在预渲染结束后自动执行 `scripts/fix-h5-build.mjs`，将 shell 内的 `/./assets/*` 改成 `./assets/*`。
- 交付前检查 `dist/client/index.html` 必须存在，且其中不得残留 `/./assets/`。

**长期约束：** 不要直接执行裸 `vite build` 作为 H5+ 交付命令；统一执行 `npm run build`，确保构建后适配步骤生效。

## 3. 组件原则

### PhoneShell

- 所有移动页面通过 `src/components/phone-shell.tsx` 统一承载状态栏、手机宽度和底部导航。
- 移动端使用 `100dvh`；桌面端最大宽度 `390px`，最大高度接近 `844px`。
- `showNav=false` 用于登录、结算、地址、会员和商品详情等独立流程页面。

### Brand

- 品牌名称固定为“山海灵感 便利店”。
- 标志和文字需要有浅色、深色两种显示状态。
- 不要临时使用其他 Logo、emoji 或网络图片替换品牌标志。

### 商品卡片

- 商品图、商品名、副标题、价格和加购按钮的顺序保持一致。
- 加购按钮使用蓝色圆形 `Plus` 图标。
- 商品名必须单行截断，副标题使用弱化灰色。
- 商品数据统一来自 `src/lib/store.ts`，不要在多个页面重复硬编码同一商品。

## 4. 页面一致性要求

- 首页、分类、发现、购物车、个人中心使用同一个底部导航组件。
- 首页“神兽新品”区域固定展示 20 个商品；修改商品数据时必须保持该区域有完整 20 项，并检查长列表滚动。
- 首页所有加购按钮必须同时完成三件事：购物车数量立即 +1、商品缩略图沿抛物线飞向底部购物车、落点图标产生一次弹性反馈。
- 当前路由必须正确高亮，购物车角标必须读取 Zustand 中的真实数量。
- 分类页保持“左侧分类 + 右侧列表”的结构。
- 购物车保持“商品列表 + 底部合计/结算”的结构。
- 发现页悬浮扫描按钮使用紧凑的 `48 × 48px` 尺寸；购物车“去结算”按钮使用 `36px` 高度，避免操作按钮视觉过重。
- 商品详情保持“大图 + 商品信息 + 规格配送 + 底部双按钮”的结构。
- 个人中心保持“用户信息 + 数值统计 + 订单入口 + 功能列表”的结构。
- 结算页保持“地址 + 配送 + 金额明细 + 提交订单”的结构。
- 设置页必须保留当前版本、清除缓存反馈和退出登录确认；个人中心顶部齿轮及功能列表中的“设置”均需进入该页面。
- “我的”菜单不包含“发票管理”，“设置”页面不包含“设备管理”。
- 收藏流程必须贯通：商品详情切换收藏状态 → 个人中心显示真实收藏数 → 收藏页可取消收藏或加入购物车。
- 订单流程必须贯通：个人中心按状态进入订单列表 → 订单列表进入详情 → 结算提交生成新订单并进入详情。
- 订单详情中的付款、确认收货操作必须更新全局订单状态，返回列表后状态保持一致。

## 5. 每次修改后的验收清单

### 代码检查

- [ ] 没有新增中文乱码。
- [ ] 没有使用 emoji 或字符代替功能图标。
- [ ] 没有为同一视觉含义引入第二套颜色或图标风格。
- [ ] 商品数据仍由 `store.ts` 提供。
- [ ] TypeScript `--noEmit` 检查通过。
- [ ] Vite 生产构建通过。

### 浏览器检查

- [ ] 在 `390 × 844` 视口检查首页。
- [ ] 检查分类页列表、分类选中态和加购按钮。
- [ ] 检查底部导航五个入口及当前项高亮。
- [ ] 检查购物车数量调整与总价。
- [ ] 检查商品详情加入购物车成功状态。
- [ ] 检查商品详情底部“购物车”四字不换行，且两个操作按钮不被挤压。
- [ ] 商品详情返回按钮必须具备至少 `40 × 40px` 点击区域，并在复杂商品图上保持清晰可见。
- [ ] 购物车商品图片、名称和价格必须可进入对应商品详情；数量加减按钮只修改数量，不触发跳转。
- [ ] 检查固定底栏没有遮挡最后一项内容。
- [ ] 浏览器控制台无新增错误。

## 6. 修改边界

- 项目连接 Lovable，禁止 force push，以及对已推送提交进行 rebase、amend 或 squash。
- 不要手动修改生成文件 `src/routeTree.gen.ts`；让 TanStack Router/Vite 生成。
- 不要为单个页面重新实现一套状态栏或底部导航。
- 不要在没有用户要求的情况下替换山海主题、商品命名或品牌文案。

## 7. 更新规则

- 每次发现新问题时，在“已出现的问题”中追加：现象、原因、解决方案和长期约束。
- 用户确认新的设计决策后，立即更新“视觉基准”或“页面一致性要求”。
- 如果实现与本文冲突，以用户最新明确要求为准，并同步更新本文，避免文档变成过时规则。
